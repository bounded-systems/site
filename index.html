Bounded Systems

Agents can write the code now. The unsolved part is keeping it honest.

One sanctioned door per kind of system power. An agent reaches git, the filesystem, a subprocess, or your tools only through it — the door checks the caller's policy, acts or refuses, and records either way. Today the highest-risk effect — git-writes — is signed and attributable to its owner.

01 / The bet

Agentic development works by bifurcation: a good idea earns its own bounded abstraction with a defined contract — and that boundary is exactly what lets you scope an agent to it.

That part is increasingly easy. The hard, unsolved part is keeping many bounded, agent-authored abstractions honest against each other as they evolve — enforcing contracts between ideas, not just within one.

Most tooling secures a single action. Bounded Systems is about the seams: every privileged effect attributable to a signed owner, one source of truth projected to many surfaces, and drift caught as a build failure so the contracts can't quietly rot.

This claim, graded: Aspirational Inter-contract enforcement is the open problem — stated as a bet, not a solved result.
02 / The model

The model, made physical

One mechanism, not twenty. A capability seam: for each kind of system power there is exactly one sanctioned way through — a door you hold a socket to, never the keys behind it. The guest-room runtime makes it physical.

room · door · guest
rooma named bundle of doors for one kind of stay. Ask for a room it doesn't know and it fails closed — a typo never widens authority.
doorone unit of authority — a unix socket to a broker daemon that holds the real keys. The guest can knock; it cannot do.
attenuationnarrowing a door by appending caveats. Append-only: authority can only ever get tighter as it is handed inward.
confinementa held capability lives only while a provider leases it, and never past that provider's ceiling.
guestthe agent the room hosts — for example claude-box, the Claude Code runtime. The engine names no guest.
# a room expands to the doors this stay holds — nothing else exists
const doors = expandRoom(rooms, catalog, "dev", env);
// narrowing only ever tightens — caveats are append-only
const narrowed = attenuate(doors[0], ["host=github.com"]);
// the rulebook is honest about what's denied, not only granted
deniedDoorSection(deniedDoors(catalog, held)) → "do not attempt"

Today prx wires these seams in-process, as library imports. The move is onto guest-room's out-of-process doors — same model, enforced across a socket — with claude-box as the first guest. Both prx and claude-box are being broken up onto that shared runtime: two tools today, converging into one door-based system.

Concretely: a room is a rootless Podman container, declared as a systemd quadlet and built reproducibly by nix; a door is a single unix socket bind-mounted in, a broker daemon holding the authority on the other side. Walls that come standard — drop-all-caps, read-only, no-new-privileges — and a short list of doors.
